Role
====
You are a senior legacy enterprise engineer tasked with generating a deliberately flawed, outdated, brittle, partially broken micro-frontend checkout application.

Objective
=========
Generate a SMALL but REALISTIC legacy micro-frontend checkout cart system using:

- React 16.14
- Legacy Redux

The application should:

- race-condition analysis,

Required Failure Modes
======================

1. Race Conditions
------------------
Include:
- duplicate OTP requests overwriting each other
- multiple checkout clicks creating duplicate orders
- stale cart totals
- coupon recalculation timing issues
- async Redux dispatch ordering bugs
- outdated state reads

Examples:
- verify OTP before latest OTP response arrives
- double-clicking "Place Order"
- cart updates during checkout

2. Direct URL Access Problems
-----------------------------
Routes should break if accessed directly.

Examples:
- /checkout/payment assumes prior route state
- /checkout/success accessible without auth
- refreshing nested routes breaks state
- deep links produce blank pages

Use:
- location.state reliance
- in-memory auth only
- missing server fallback routing

3. Resource Access Vulnerabilities
----------------------------------
Include:
- order lookup by sequential numeric ID
- missing authorization checks
- frontend-only admin protection
- cart retrieval without ownership validation
- predictable OTP session IDs

Examples:
GET /api/orders/12

returns another user's order.

4. Broken State Management
--------------------------
Redux state should be:
- deeply nested
- duplicated
- inconsistent
- partially mutable

Example:
state.checkout.cart.current.active.items.list

Include:
- direct state mutation
- reducers with side effects
- non-serializable state
- promises inside Redux
- Date objects in state

5. Coupon System Brittleness
----------------------------
Coupons should:
- sometimes apply twice
- fail under rapid clicks
- allow negative totals
- use floating-point math errors
- trust frontend calculations

Examples:
- coupon stack race conditions
- coupon removed after refresh
- stale totals shown to user
- backend trusting client subtotal

6. OTP System Problems
----------------------
OTP system should include:
- hardcoded OTP bypass
- OTP stored in Redux
- OTP stored in localStorage
- no rate limiting
- resend race conditions
- expired OTP still accepted sometimes
- multiple active OTP sessions

Examples:
OTP:
123456

admin bypass:
999999

7. Security Vulnerabilities
---------------------------

A. XSS
------
Render coupon messages or product names using:
dangerouslySetInnerHTML

without sanitization.

B. Insecure Storage
-------------------
Store:
- JWT
- OTP session
- cart state

inside localStorage.

C. Sensitive Logging
--------------------
Log:
- JWT tokens
- OTP values
- request payloads
- postgres errors

to console.

D. Trusting Frontend
--------------------
Backend should trust:
- frontend prices
- frontend totals
- frontend coupon values

E. Missing Validation
---------------------
Minimal validation:
- negative quantities possible
- malformed phone numbers accepted
- invalid coupon formats crash app
